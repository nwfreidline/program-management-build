## Severity Definitions

| Severity | Definition | Response Time | Escalation |
|----------|-----------|---------------|-----------|
| **Critical** | Project stopped or major deliverable at risk; no workaround | [Immediate / Same day] | [Sponsor / Steering Committee] |
| **High** | Significant impact on schedule, cost, or quality; limited workaround | [24 hours] | [Project Manager / Sponsor] |
| **Medium** | Moderate impact; workaround available | [48 hours] | [Project Manager] |
| **Low** | Minor impact; does not affect critical path or deliverables | [1 week] | [Team Lead] |

## Issue Categories

| Category | Description | Examples |
|----------|-------------|---------|
| **Technical** | Technology, system, or tool issues | System outage, integration failure, defect |
| **Resource** | People, skills, or availability | Key person unavailable, skill gap, turnover |
| **Process** | Workflow, procedure, or methodology | Approval delays, unclear requirements |
| **External** | Vendor, regulatory, or environmental | Vendor delivery late, regulation change |
| **Scope** | Requirements or deliverable issues | Scope creep, requirement conflict |
| **Communication** | Stakeholder or information issues | Miscommunication, missing information |

## Issue Management Process

1. **Identify** — Issue reported by any team member
2. **Log** — Assign ID, document details, set severity
3. **Assign** — Allocate to owner with target resolution date
4. **Investigate** — Determine root cause and resolution options
5. **Resolve** — Implement solution
6. **Verify** — Confirm issue is resolved
7. **Close** — Document resolution, update log
